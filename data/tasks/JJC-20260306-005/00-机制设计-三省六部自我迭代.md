# JJC-20260306-005｜Edict 三省六部制「自我迭代进化」机制设计（可落地版）

> 目标：让制度本身能“自己变好”，不靠皇上情绪/人工盯人。
> 设计原则：小闭环、高频率、证据化、可回滚、可度量。

---

## 0. 一句话总纲
**用指标触发（Metrics）→ 任务化（JJC）→ 三省审议（Zhongshu/Menxia/Shangshu）→ 验收证据（Evidence）→ 复盘提案（RFC）→ 版本化发布（CHANGELOG）→ 自动巡检回归（Cron）**，形成循环。

---

## 1) 机制的“产出物”标准（没有这些=没有进化）
每一轮迭代必须产出四件套：
1. **问题清单**（来自指标/巡检/反馈）
2. **改进提案 RFC**（最小决策包）
3. **落地变更**（脚本/规则/配置/文档/模板）
4. **验收证据**（命令输出/截图/PR/对比指标）

统一归档路径：
- `data/tasks/JJC-YYYYMMDD-NNN/`（每轮一个任务目录）

---

## 2) 指标体系（Metrics）——用数据逼出改进
### 2.1 必须监控的 10 个核心指标（建议作为 Dashboard 顶部摘要）
1. **未完成任务数**（state!=Done）
2. **停滞任务数（不含明确阻塞）**（`updatedAt > 6h/24h` 且 `block=无`）
3. **明确阻塞任务数**（`block!=无`，用于“等待皇上/外部依赖”不算停滞，但必须可见）
4. **调度/派发错误数**（`_scheduler.lastDispatchStatus=error` 或 lastDispatchError 非空）
5. **超时审议数**（中书/门下超 SLA）
6. **Done 无证据数**（Done 但缺验收证据路径/命令）
7. **回滚次数**（本周 revert 或“撤回/重做”次数）
8. **巡检有效率**（巡检发现问题->进入任务->完成 的比例）
9. **工具错误率**（如 opencode tool JSON parse error 发生次数）
10. **人类打断率**（需要皇上介入的次数/原因分类）

### 2.2 指标触发的“硬规则”（触发=必须立项）
- 任何 **P0**（主链路断：看板不可用、任务源写不了、渠道不可达）→ 立即立项修复
- **停滞任务数 ≥ 3** 或 **任一任务停滞 > 24h** → 必须发起“阻塞清理”迭代
- **Done 无证据数 > 0** → 必须发起“验收制度修复”迭代
- **工具错误率** 连续两天>阈值（比如>10次/天）→ 必须发起“工具稳定性专项”

---

## 3) 角色分工（谁负责让制度变好）
### 3.1 太子（总控/入口）
- 收集：反馈、巡检、指标异常
- 立项：创建 JJC 迭代任务（TOP1 可中断其它任务）
- 追踪：每日 2 次（早/晚）确认“是否推进到下一阶段”

### 3.2 中书省（提案者）
- 产出 RFC：目标/范围/验收/风险/回滚
- 拆分 TODO：每条 TODO 必须可验收

### 3.3 门下省（审议者/质量门禁）
- “三问审议”：
  1) 目标是否清晰可验收？
  2) 是否最小改动、可回滚？
  3) 是否会引入新复杂度/新噪音？
- 给出结论：准/不准/补充材料

### 3.4 尚书省（落地者/执行验收）
- 做变更并提供证据
- 把证据写入任务目录与看板 progress

---

## 4) 工作流（让进化自动发生）
### 4.1 每周固定节奏（推荐）
- **周一**：制度健康度盘点（指标汇总 + 问题排序）
- **周三**：专项修复窗口（工具稳定性/看板可靠性）
- **周五**：复盘与发布（RFC 合并、CHANGELOG、下周计划）

### 4.2 每日最小节奏（必须）
- 早：巡检摘要（自动）→ 若触发阈值 → 自动立项或人工立项
- 晚：验收清点（Done 是否有证据、是否可复现）

---

## 5) RFC 模板（最小决策包，门下省审议对象）
文件：`data/tasks/<JJC>/rfc.md`

必须字段：
- 背景（触发指标/故障/反馈）
- 目标（1句话）
- 范围（in/out）
- 方案（不超过 1 页）
- 验收（命令/截图/对比指标）
- 风险与回滚（必须）
- 负责人（中书/门下/尚书分别是谁）

---

## 6) 验收制度（把“完成”变成客观事实）
### 6.1 Done 的硬条件（缺一不可）
- 交付物路径已落盘（output/evidencePath）
- 验收命令可复制运行
- 关键输出/截图已归档
- 回滚方案存在（哪怕是 git revert / 恢复旧配置）

### 6.2 Block（阻塞）字段的制度化
- `block=无`：表示“可推进”，停滞阈值对其生效（6h/24h）。
- `block!=无`：表示“明确外部依赖/等待皇上/等待三方”，**不计入停滞**，但必须在健康报告中单独统计与点名。
- 任何 `block!=无` 的任务必须写清：**谁来解除、需要什么、预计何时解除**。

### 6.2 典型反例（禁止）
- 只有“写了文档”但没有练习/回执（培训类）
- 只有“跑通一次”但没有自动化回归（稳定性类）

---

## 7) 自动化落地点（立即可做的 4 个改动）
### 7.1 新增“制度健康度”定时任务
- cron：每天 2 次（例如 10:00 / 18:00）
- 内容：读取 `tasks_source.json`，计算 8 指标，写入 `data/health_report.json` 并在看板显示。

### 7.2 新增“验收证据检查”脚本
- 扫描 Done 的任务目录是否有 evidence 字段/文件
- 若缺：自动把任务标红（写 progress）

### 7.3 约束工具调用输出（减少 JSON parse）
- 在团队提示词/工作流里写死：工具参数一次只允许一个 JSON 对象；批量拆多次调用。

### 7.4 环境一致性（解决“openclaw not found / PATH”类派发失败）
- systemd 启动的 Dashboard/Loop 进程常见 PATH 不含 nvm 的 bin，可能导致 `openclaw` CLI 找不到，进而引发 `_scheduler.lastDispatchError`。
- 规约：所有需要调用 `openclaw` 的服务必须显式设置 PATH 或使用绝对路径。
