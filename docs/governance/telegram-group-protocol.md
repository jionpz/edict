# Telegram 朝会群协作协议（极简版）

> 目标：让皇上在群里“@谁谁答”，并让大臣在办事过程中需要协作时，能在群内通过 @ 对方完成沟通。
> 适用范围：Edict 三省六部制在 Telegram 的朝会群。

## 0. 术语
- **朝会群**：三省六部统一协作群（Telegram 群聊）。
- **大臣 bot**：各官署对应的 Telegram bot（如 `@zhongshu_*_bot`）。

## 1) 皇上发言规则（最少规则，最高执行力）
1. 皇上在群里发言：**@哪位大臣，哪位必须回复**；未被 @ 的大臣不得插话。
2. 群内默认 `requireMention=true`：只有被 @ 的 bot 才会响应，避免刷屏。

## 2) 大臣回复规范（让对话可验收、可追踪）
每次回复必须满足：
- **先结论（1 句）**
- **再证据/路径/下一步（≤5 行）**
- 长内容必须落盘（PR/commit/文件路径/链接）。

禁止：只说“已处理/已完成”但不提供证据。

## 3) 大臣互通制度（核心：办事过程中需要协作必须@对方）
凡大臣在自己工作的过程中需要别的官署信息/决策/产出，必须在群内发起协作请求，并 @ 对方 bot。

### 3.1 协作请求模板（强制 4 行）
```
@对方_bot
任务ID: JJC-xxxx（无则 N/A）
我需要: <一句话需求/输入>
截止: <时间>（请按“结论+证据/链接/路径”回复）
```

### 3.2 被@的大臣回复要求
- 必须在截止前回复。
- 如需第三方：按同模板继续 @ 下一位大臣。

## 4) 解决 "no access to message" 的必要条件
### 4.1 机器人隐私模式（BotFather）
为保证 bot 能读取群内上下文（尤其是引用/回复链），需在 BotFather 对所有大臣 bot 执行：
- `/setprivacy` → 选择 bot → `Disable`

否则会出现：bot 无法访问被回复/被引用消息，进而提示 `no access to message`。

### 4.2 操作建议（规避法）
- 尽量不要用“回复某条消息再@机器人”传递上下文。
- 建议在新消息中把上下文直接写全再@。

## 5) 安全边界建议
- 群接入建议使用 allowlist：只允许朝会群进入系统。
- 对 bot 的 `allowFrom` 建议仅放皇上（以及明确授权的管理员）。
