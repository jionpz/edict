# 全员培训包：Codex / Claude Code / OpenCode（AI 代码助手）速查与练习

> 目标：让“百官”在 30–60 分钟内具备
> 1) 能把需求拆成可执行任务
> 2) 能驱动代码助手修改项目并提交
> 3) 知道什么时候该用、什么时候不该用
> 4) 碰到常见坑（尤其 WSL/浏览器回调）能自救

---

## A. 统一心法（先讲清楚）

### 1) 你不是在“聊天”，是在“下达工单”
给 AI 代码助手的输入，必须包含：
- **目标**：一句话要达到什么状态
- **边界**：哪些不能动、哪些必须保留
- **验收标准**：用什么命令/页面/截图证明成功
- **上下文**：仓库路径、模块名、相关文件、现有错误日志

推荐模版：
```
目标：……
约束：……
环境：OS/WSL/Node/Python 版本……
现状：现在卡在哪（日志/截图/复现步骤）……
期望：最终我能用这些步骤验收：
  1)
  2)
输出要求：给出改动文件清单 + 关键命令 + 可能风险
```

### 2) 三段式协作流程（最稳）
1. **先让它“读代码/复述理解”**（防止盲改）
2. **再让它“给方案 + 变更点清单”**（你确认方向）
3. **最后让它“动手改 + 提交 + 验收命令”**

### 3) 何时不该用
- 需求不清楚：先澄清再开工
- 涉及密钥/生产数据：AI 只看脱敏材料
- 复杂大改：先拆小步、逐步提交

---

## B. 工具选择：Codex / Claude Code / OpenCode 怎么用

> 你们内部叫法不同，但本质都属于“在终端/IDE 里跑的编码代理”。

### 1) 适用场景对照
- **Codex**：修 bug、写小功能、重构、写测试、读大项目（速度快）
- **Claude Code**：更擅长复杂推理/长上下文，适合设计方案、跨文件改动
- **OpenCode**：以你们内部实际为准（若是 OpenClaw 的 ACP/编码 harness，它就是“在聊天里下达编码任务”的管道）

### 2) 最小练习路线（从易到难）
- 练习 1：改一个文案/README（验证提交链路）
- 练习 2：加一个 CLI 参数（验证参数解析/帮助文档/测试）
- 练习 3：修一个可复现的报错（从日志定位到代码）
- 练习 4：小重构（拆函数/加类型/加注释），保证测试通过
- 练习 5：加一个“健康检查/状态页”小功能（端到端）

每次练习必须产出：
- PR/commit
- 变更文件列表
- 验收命令

## C. 在你当前架构（Windows + WSL2）下的关键注意事项

### 1) WSL 里网页登录回调打不开（你已经遇到）
原因：工具在 WSL 起了 `http://127.0.0.1:xxxxx` 回调，但浏览器在 Windows 打开，Windows 的 127.0.0.1 不是 WSL。

**推荐解决法**（按优先级）：
1. **用 PAT/粘贴 token 登录**（最稳，不依赖浏览器回调）
2. **用 Device Code 登录**（终端给 code，去网页输入）
3. 再考虑折腾 localhost 端口转发（不推荐作为日常方案）

### 2) 代码/服务跑在哪里要一致
- 你已选择“原生在 WSL 跑”三省六部：
  - `bash scripts/run_loop.sh`
  - `python3 dashboard/server.py`（7891）
- 不要半套在容器、半套在 WSL（会造成“看得见、点不动、控制割裂”）

## D. 统一验收清单（百官每次交付都要对照）
- [ ] 我能用 3 句话复述目标与边界
- [ ] 我能给出复现步骤/验收步骤
- [ ] 我能列出改动文件清单
- [ ] 我能给出至少 1 个回滚方案
- [ ] 我没有把 key/隐私数据粘给 AI

---

## E. 给百官的“默认指令”范例（可直接复制给代码助手）

### 范例 1：修 bug
```
请先阅读并复述你将修改的文件与原因。
然后给出最小修复方案（含风险与回滚）。
我确认后再实施。
验收方式：提供具体命令与预期输出。
```

### 范例 2：加功能
```
目标：新增 XXX 功能。
约束：不改动现有 API；保持向后兼容。
验收：运行 `...` 通过；页面 `...` 展示正确。
输出：改动文件清单 + 提交信息建议。
```
