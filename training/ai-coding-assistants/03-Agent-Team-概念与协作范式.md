# Agent Team 概念与协作范式（给百官）

## 1. 什么是 Agent Team
**Agent Team** 指的是：把多个“专长不同的代理/工具”组织成一个团队，按流程协作完成交付。

在三省六部语境下可以对应：
- **太子（入口/分拣/把关）**：接旨、澄清、拆解任务、控制节奏与风险
- **中书省（方案/起草）**：把模糊旨意写成可执行方案（步骤、验收、风险）
- **尚书省（实施/交付）**：真正改代码、跑命令、修 bug、出产物
- **门下省（审议/质量）**：审查方案与交付，保证不偏离目标

核心不是“多开几个 AI”，而是：
- **分工**（不同角色负责不同产出）
- **接口**（每一步有明确输入/输出）
- **验收**（每个角色都要给可验证证据）


## 2. 为什么需要 Agent Team（而不是单一代码助手）
单一代码助手容易出现：
- 盲改、跑偏、缺少验收闭环
- 修改范围过大，回滚困难
- 没人负责“澄清需求/风险控制/质量审查”

Agent Team 用流程把风险压住：
- 先把“旨意”变成“工单”（中书省）
- 再把“工单”变成“可合并的变更”（尚书省）
- 最后做“审议与验收”（门下省/太子）


## 3. 标准协作流程（建议所有任务都用）
### Step 0：太子分拣
- 输出：一句话目标 + 是否需要建 JJC + 交付物清单

### Step 1：中书省起草执行方案
- 输出：
  - 任务拆解（3-7 个子步骤）
  - 风险点（权限/外发/数据安全）
  - 验收脚本（命令/页面/截图）

### Step 2：尚书省实施
- 输出：
  - 变更文件清单
  - commit/PR
  - 验收结果（命令输出、截图）

### Step 3：门下省审议
- 输出：
  - 是否符合目标/是否越权
  - 是否需要回滚/补测

### Step 4：太子回奏皇上
- 输出：
  - 摘要（1段话）
  - 证据（链接/路径/命令）
  - 阻塞项（需要皇上拍板的点）


## 4. 训练用“角色提示词”模板（百官自用）
- **中书省**：
  - “把需求写成可执行 SOP：步骤、输入输出、验收、风险、回滚。”
- **尚书省**：
  - “严格按 SOP 改代码；每步提交；不给大段猜测；输出可复现证据。”
- **门下省**：
  - “审阅变更：找边界问题、权限/安全风险、验收是否充分。”


## 5. 最小化 token/成本的团队协作
- 先让中书省产出 1 页方案，皇上/太子确认后再让尚书省动手
- 每次只传“必要上下文”（文件路径+关键片段），不要整仓库贴
- 只在异常/争议时展开长推理
